Task System Documentation

Description

This is a system for organizing various general tasks using a manual-first approach and designed to last for life. Parsers can be used on top of it for a better overview, but must reflect manual changes done to files.

Folder Tree

This table describes the root folder tree. It is your choice how you want to organize within them. The root folder tree is not a requirement, just a suggestion.

Folder Description
archive Archive of completed or abandoned tasks, which are moved here.
docs This documentation. It is best that documentation follows the task system over its lifetime. Things may change.
store Open tasks. The main store of tasks that are ongoing, recurring or just ideas.
templates Various templates for creating new items, ready to copy from.

Task

A task is simply a folder with a metadata.txt file inside. The task folder may have other arbitrary files and sub folders relevant to the task.

Generally you would want a main document describing the task. This could be a simple text file or a more advanced format.

The state of the task is reflected by its location in the folder structure. Completed or abandoned tasks are meant to be moved away to an archive. You could have state in the metadata.txt but it may be redundant and easy to get out of sync when working manually. Therefore, the state of the task is easier to represent by its location in the folder structure. You are meant to move the task folder out of the way once you are done with it (archive or delete).

Metadata.txt

Each task should have a file that describes task details. This makes it possible to destinguish task folders from category folders in the structure. It is a simple case insensitive format that is easy to read, write and parse. Lines starting with the hash character ("#") are treated as comments and should be ignored by the parser.

All attributes are optional to tasks, but there is a metadata-defaults.txt file in the templates folder with some default values if an attribute is missing.

Attribute Type Description
Version Text Metadata format version. This is mainly for future proofing and make it easier for parsers. Semantic version numbers are recommended. It is best to have this as the first attribute.
Title Text Short title for the task, suitable for use in lists.
Summary Text A brief summary that is useful when browsing through tasks.
Main-File Text Name of which file that is considered the entry point of the task, in case there are multiple documents. Typically the name of the file would indicate it and this attribute is not needed. The attribute can also provide a hint to parsers.
Created Date Task creation date. ISO 8601 format is best here to make it easier for parsers, but it could be any format.
Due-Date Date When the task is due, if relevant.
Recurring Flag Whether the task is done more than once. Typically "yes" or "no".
Recurring-Period Period How often the task is repeated (such as "weekly" or "yearly"), if relevant. This attribute may be redundant depending on how your structure your store.
Tags List Comma separated list of arbitrary tag names. Useful for searching or sorting.

You can add your own attributes on a need-to basis to aid parsers in presenting or sorting tasks. Though, the parser must be made to know about your custom attributes.

Example

This shows a small metadata.txt file with attributes relevant to the task. Other attributes are left out.

Version: 1.0
Title: Task Title
Summary: A brief summary.
Created: 2025-01-30
Tags: example, simple-task

Sensitive Tasks

Some tasks may have content (such as medical records) that may be too sensitive to store in plain text. This task system does not specify how sensitive material should be handled, but provides some ideas on how to use it in combination with the task system. In metadata.txt, you can mark the task as encrypted. Either the task content could be in this in encrypted form, or you can refer to another location.

Using a password manager, such as KeePassXC, is highly recommended. You can store the encryption keys or passwords in that database. Optionally you can also refer to the password database in metadata.txt for unlocking the content.

For short-term tasks you can work with the task unencrypted, and then encrypt when archiving the task. A password protected ZIP file may be sufficient, with the password stored in the password database.

All this depends on your risk tolerance and how the task system is stored. How secure you want it to be depends on you. Encryption is out of scope for the task system itself.

Task Store

This folder contains all tasks and is the root of the category folder hierarchy. Task folders are placed where suitable.

The folder structure is largely arbitrary and user defined. This allows for organizing tasks as the user decides. This Git repository contains an example of one of many possible structures.

Parsers will use the structure to represent the state or status of the task, depending on how you organize it.

Templates

Contains templates to make task creation easier. It has a folder template for a new task with a basic template with metadata.txt and task.md set up.

The metadata-default.txt file has some default values to aid in parsing. It contains placeholder values for attributes to make it possible to list the task.
